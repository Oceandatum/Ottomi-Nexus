# 贡献指南 Contribution Guidelines

欢迎参与本项目的贡献！在提交 **Issue（问题反馈）**、**Pull Request（PR，代码合并请求）** 或任何其他形式的贡献之前，请仔细阅读本指南。

**重要提示**：本项目采用**差异化商业授权模式**（详见 [LICENSE](./LICENSE) 与 [NOTICE](./NOTICE) 文件）。**所有贡献行为均视为您已充分理解并同意本指南及相关授权协议中的全部条款。**

------

## 📌 一、 贡献范围与权限 (Scope & Permissions)

本项目的贡献管理严格区分以下两类贡献：

### 1. **公共开源贡献 (Public Open Source Contributions)**

任何个人或组织均可对本项目**公共、非核心**的功能模块（如文档翻译、通用工具类优化、非业务逻辑的代码注释修正等）进行贡献。

- **授权协议**：此类贡献将默认采用 **Apache License 2.0** 协议进行授权。
- **目的**：提升项目的公共可用性与透明度。

### 2. **商业 / 专有贡献 (Commercial/Proprietary Contributions)**

任何旨在修改、扩展、集成本项目**核心商业功能**、**底层架构**或**专有模块**的贡献，均属于商业专有范畴。

- **严格禁止**：未经**上海奥腾计算机科技有限公司**（以下简称 “权利人”）书面商业授权，任何个人或组织不得将此类贡献用于商业售卖、再授权、内部转售或任何盈利性行为。
- **强制要求**：所有商业专有贡献必须由权利人官方团队审核、接受并最终合并。

## 🚫 二、 禁止行为 (Prohibited Actions)

为保护项目的知识产权与商业授权体系，以下行为被**严格禁止**：

1. **核心模块绕过**：禁止通过修改、插件或二次开发的方式，**规避、替换或绕过**本项目的核心授权验证机制、版权声明或使用限制。
2. **逆向工程滥用**：禁止对本项目（除明确标识为开源的第三方组件外）进行**反编译、反向工程、反汇编或反汇编**。
3. **知识产权侵权**：禁止将本项目的任何部分（包括但不限于代码、架构、设计、文档）**复制、分发、二次授权**给第三方用于商业目的。
4. **第三方组件冲突**：禁止在贡献代码中引入与本项目已集成的**第三方开源组件**（如 DataEase、Apache DolphinScheduler 等，详见 NOTICE）协议相冲突的代码。

## 🧾 三、 Issue（问题反馈） 创建规范

如果您发现了 bug 或有功能建议，请通过 Issue 反馈。

### 1. Bug 报告 (Bug Report)

请确保您的报告包含以下信息，以帮助我们快速定位问题：

- **清晰的标题**：简述问题核心。
- **复现步骤**：详细描述如何复现该 bug。
- **预期行为**：您期望系统如何运行。
- **实际行为**：系统当前的错误表现。
- **环境信息**：操作系统、浏览器版本、设备型号等。
- **附件**：如有必要，请附上截图、日志文件或录屏。

### 2. 功能建议 (Feature Request)

- **清晰的标题**：明确建议的功能名称。
- **详细描述**：说明该功能的使用场景、解决的问题以及期望的实现效果。
- **兼容性说明**：请说明该建议是否会影响现有商业授权用户的权益。

## 🔄 四、 Pull Request（代码合并）流程

如果您希望提交代码修改，请遵循以下流程：

### 1. 前置检查 (Pre-flight Checks)

在提交 PR 前，请务必确认：

- 您已阅读并理解 [LICENSE](./LICENSE) 与 [NOTICE](./NOTICE) 的全部条款。
- 您的贡献**不属于**禁止行为列表中任何内容。
- 您的代码**未包含**任何受版权保护的、未授权的第三方代码或资源。

### 2. 提交规范 (Commit Message Guidelines)

请使用清晰、标准的 commit 信息，便于项目维护者审核。

格式如下：

plaintext

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

- **type（类型）**: 必须是 `feat` (新功能), `fix` (bug 修复), `docs` (文档), `style` (格式调整), `refactor` (重构), `test` (测试), `chore` (构建 / 辅助工具) 之一。
- **scope（范围）**: 说明修改的模块，例如 `core`, `ui`, `utils`, `docs`。

**示例**：

```
fix(core): 修复了用户认证模块中的一个竞态条件 (Race Condition)
```

### 3. 代码风格与规范 (Code Style & Standards)

- 所有贡献代码必须遵循项目已定义的编码规范、命名约定与注释标准。
- 禁止提交包含未使用的废弃代码、注释掉的调试代码或无关的格式化更改。
